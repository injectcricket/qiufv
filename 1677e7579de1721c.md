# ResourceBridge

ResourceBridge 是一个面向技术团队与独立开发者的外链资源归集与导航系统。项目定位为轻量级的技术文章、代码示例与工程实践的外部链接聚合平台，帮助用户从分散的博客、文档与社区帖子中快速定位高价值技术内容。

本项目不存储任何实质性的文章内容或代码文件，仅提供 URL 索引与分类导航功能。目标用户包括技术负责人、文档维护者、知识管理工程师以及日常需要查阅大量技术资料的一线开发人员。ResourceBridge 通过结构化的链接管理、标签过滤与全文检索功能，将零散的外部信息转化为可检索、可追溯、可分享的内部知识资产。

ResourceBridge 适用于以下场景：团队内部技术周报的素材采集、项目文档中的参考链接归档、新人入职培训时的资料索引编制，以及个人开发者对技术博客的长期追踪与管理。项目采用纯静态页面生成方案，无需数据库依赖，可部署于任何支持 HTTP 服务的环境。

## 功能概览

**结构化链接目录** 基于文章分类与编号自动生成多级导航树，支持按年份、主题、来源站点筛选。

**全文检索引擎** 对链接标题、摘要与标签字段进行倒排索引构建，支持布尔查询与模糊匹配。

**批量导入接口** 提供 CSV 与 JSON 格式的链接批量导入工具，兼容主流书签导出数据。

**标签管理系统** 支持自定义标签的创建、合并与重命名，标签层级深度不限。

**访问状态监控** 定时发起 HTTP HEAD 请求检测链接可用性，输出失效链接报表。

**元数据扩展字段** 每条链接可记录阅读状态、重要等级、归档日期与负责人信息。

**响应式输出模板** 生成适配桌面与移动终端的 HTML 页面，无外部依赖，加载迅速。

**RSS 订阅输出** 按链接更新时间生成 RSS 2.0 格式 Feed，支持第三方阅读器订阅。

## 应用场景

技术团队内部知识库建设。团队文档管理员将项目开发过程中参考的技术博客、官方 API 文档、问题排查帖子统一归集到 ResourceBridge 中，并打上对应模块标签。新成员加入时可通过标签快速学习相关领域的前置资料，减少重复沟通成本。

个人技术博客的参考链接管理。独立博主在撰写技术文章时，将引用的外部链接统一录入 ResourceBridge，自动生成文章末尾的参考列表。当外部链接失效时，监控功能会提前预警，博主可及时更新或替换引用源。

技术培训课程的配套资料索引。培训讲师将课程涉及的延伸阅读材料、视频教程地址、在线代码演示平台全部整理为链接集合，学员通过 ResourceBridge 的目录树按课时顺序访问，无需记忆大量零散网址。

开源项目的贡献文档引导。开源项目维护者将贡献指南、编码规范、设计文档的外部参考链接集中存放于 ResourceBridge，外部贡献者提交代码前可先查阅相关背景资料，降低沟通门槛。

## 快速开始

以下命令序列适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 运行环境。

```bash
git clone https://github.com/resource-bridge/resource-bridge.git
cd resource-bridge
npm install
npm run build -- --input ./data/links.json --output ./dist
```

执行完成后，`./dist` 目录下生成完整的静态站点文件，可使用任意 HTTP 服务器托管。开发模式下可使用 `npm run dev` 启动本地预览服务，默认监听 3000 端口。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高 | 是 | 运行时环境，用于执行构建脚本与本地服务 |
| npm 9.x 或更高 | 是 | 包管理器，用于安装项目依赖 |
| Git 2.30 或更高 | 是 | 版本控制工具，用于克隆仓库与拉取更新 |
| curl 或 wget | 否 | 可选，用于通过命令行测试链接可用性 |
| Python 3.9+（仅开发依赖） | 否 | 若需运行自定义导入脚本，需 Python 环境 |
| SQLite 3（仅高级模式） | 否 | 若启用本地缓存数据库，需 SQLite 支持 |

## 贡献指南

贡献者请遵循以下流程提交变更。所有贡献均需遵守行为准则，并在提交前通过完整的测试套件。

第一步，从主仓库 Fork 项目到个人账户，使用 Git 克隆本地副本。创建新分支时请使用 `feature/` 或 `fix/` 前缀，分支名称应简要描述变更内容。

第二步，安装开发依赖并启动本地构建。运行 `npm install` 安装所有依赖包，执行 `npm run test` 确保现有测试全部通过。新增功能必须附带对应的单元测试或集成测试。

第三步，在 `data/links.json` 中按规范添加或修改链接数据，确保所有新增 URL 均为有效技术资源。若涉及标签变更，需同步更新 `data/tags.json` 文件。

第四步，提交代码前执行 `npm run lint` 进行代码风格检查，修复所有报告的问题。提交信息格式遵循 Conventional Commits 规范，即 `type(scope): description` 模式。

第五步，推送分支至远程仓库并创建 Pull Request。PR 描述中需明确说明变更目的、影响范围以及测试覆盖情况。维护者将在三个工作日内审核并给出反馈。

## 常见问题

Q: 项目是否支持 HTTPS 协议的链接导入？导入时是否会自动转换协议格式？

A: 支持导入任意协议的链接，包括 HTTPS、HTTP、FTP 等。系统不会自动转换或修改用户输入的协议部分，完全保留原始 URL 格式。若需要统一协议，用户可在导入前自行预处理数据文件。

Q: 生成的静态页面中链接跳转是否会经过中间页或添加跟踪参数？

A: 不会。ResourceBridge 生成的页面中所有外部链接均为直接跳转，不添加任何重定向中间页、跟踪参数或点击统计脚本。项目承诺不收集任何用户点击数据，所有链接按原始地址原样输出。

Q: 如何迁移已有书签数据到 ResourceBridge？

A: 项目提供了 `tools/import-from-html.js` 脚本，可解析浏览器导出的 HTML 书签文件。执行 `npm run import -- --source bookmarks.html --output data/links.json` 即可完成转换。也支持从 Pocket、Instapaper 等服务的 CSV 导出格式进行导入。
