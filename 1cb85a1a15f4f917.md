# TechResource Hub

TechResource Hub 是一个面向开发者、技术研究者与开源爱好者的高质量技术文章与教程聚合索引项目。项目系统化收集并分类来自 blog.ityiqv.cn 的深度技术文章，涵盖编程语言、算法与数据结构、系统设计、数据库原理、网络协议、前端工程化、DevOps 实践等多个核心计算机科学领域。

本项目定位为技术外链汇总与知识导航站，不直接存储或托管任何文章内容，而是通过结构化索引与分类标注，帮助开发者快速定位到具体技术问题的解决方案与原理剖析。目标用户包括初级开发者（寻找入门教程）、中级工程师（深入理解底层机制）以及技术团队负责人（评估技术选型与最佳实践）。

## 功能概览

**结构化文章索引**：按照编程语言、框架生态、基础设施、算法理论、工程效能等维度对每一条收录链接进行多标签分类，并提供按标签筛选的视图。

**全文元数据检索**：基于文章标题、编号、发布日期与摘要字段构建轻量级检索能力，支持关键词模糊匹配与精确 ID 查询。

**批量导入与自动归档**：支持通过 CSV 或 JSON 格式批量导入新的文章链接，系统自动提取 URL 中的文章 ID 并生成归档时间戳。

**阅读进度追踪**：为注册用户提供文章标记已读、添加个人笔记与收藏夹功能，阅读进度数据本地持久化存储。

**每日技术早报**：每日 UTC 0 点从索引中随机抽取 5 篇不同类别文章生成早报推荐列表，帮助开发者保持技术视野更新。

**外部源健康检查**：定时对索引中的全部外链发起 HEAD 请求，检测链接可用性状态，并在管理面板中标记失效链接。

**标签云与热度视图**：根据文章被点击与收藏频次动态生成标签云，高频标签字体放大展示，便于发现当前社区关注的热点主题。

**导出为 Markdown 目录**：支持将当前筛选或全部索引导出为结构化 Markdown 清单文件，便于离线阅读或嵌入其他文档体系。

## 应用场景

**场景一：新项目技术选型调研**  
团队启动新后端服务项目时，架构师可通过本索引快速检索关于微服务框架对比、RPC 协议选型与容器化部署的系列文章，集中阅读多方观点以辅助决策。

**场景二：面试前系统复习**  
求职者在准备算法与系统设计面试时，可利用本项目的标签体系按「数据结构」「动态规划」「分布式一致性」等标签筛选文章，形成体系化的复习路径。

**场景三：技术文档写作参考**  
技术博主或开源项目维护者在撰写技术方案说明时，可通过本索引查找同类问题的已有剖析文章，参考其论证结构与代码示例组织方式，提升文档质量。

**场景四：新人入职培训知识包**  
团队为新入职开发人员定制学习计划时，可从索引中挑选出关于版本控制规范、代码评审流程、CI/CD 流水线配置等基础类文章，形成标准化的培训材料包。

**场景五：技术分享会选题挖掘**  
内部技术分享组织者可通过浏览标签云与热度视图，发现近期讨论热度上升的技术主题，据此策划下一期分享会的内容方向。

## 快速开始

以下指令帮助您在本地环境快速启动 TechResource Hub 的开发实例。

```bash
# 克隆项目仓库至本地
git clone https://github.com/techresource-hub/techresource-hub.git

# 进入项目根目录
cd techresource-hub

# 安装项目依赖（使用 npm）
npm install

# 启动开发服务器，默认监听 http://localhost:3000
npm run dev
```

启动成功后，打开浏览器访问本地地址即可浏览文章索引。首次启动将自动执行种子数据初始化，包含本批次全部 250 条文章链接的元数据预生成。

## 项目结构

项目采用分层架构设计，前端基于 React + TypeScript，后端基于 Node.js + Express，数据层使用 SQLite3。以下为项目核心目录结构及说明。

```
techresource-hub/
├── frontend/                          # 前端应用目录
│   ├── src/
│   │   ├── components/                # UI 组件库（Button, Card, TagCloud 等）
│   │   ├── pages/                     # 页面级组件（Home, Search, Archive）
│   │   ├── hooks/                     # 自定义 React Hooks（useFetch, useLocalStorage）
│   │   ├── utils/                     # 前端工具函数（URL 解析、日期格式化）
│   │   └── styles/                    # 全局样式与主题变量
│   ├── public/                        # 静态资源（favicon, robots.txt）
│   └── package.json                   # 前端依赖声明
├── backend/                           # 后端服务目录
│   ├── src/
│   │   ├── routes/                    # RESTful API 路由（articles, tags, health）
│   │   ├── controllers/               # 请求控制器（业务逻辑编排）
│   │   ├── models/                    # 数据模型（Article, Tag, User）
│   │   ├── services/                  # 核心服务（索引构建、健康检查）
│   │   ├── middleware/                # 中间件（日志、鉴权、错误处理）
│   │   └── workers/                   # 后台任务（早报生成、链接探测）
│   └── package.json                   # 后端依赖声明
├── database/                          # 数据库相关
│   ├── migrations/                    # 表结构变更脚本（按时间戳命名）
│   ├── seeds/                         # 种子数据（本批次 250 条文章初始数据）
│   └── techresource.db                # SQLite3 数据文件（运行时生成）
├── docs/                              # 项目文档（见文档导航表格）
├── scripts/                           # 运维脚本
│   ├── health-check.sh                # 链接健康检查 shell 脚本
```

## 贡献指南

我们欢迎并感谢任何形式的贡献，包括但不限于新增文章链接、修复错误链接、完善文档、提出功能建议或改进代码实现。请遵循以下步骤参与贡献。

第一步：阅读项目行为准则与贡献者公约，确保提交内容符合社区规范。所有交互需保持专业与友善。

第二步：从 Issue 列表中选择或自行创建 Issue 描述待解决问题或新增功能。建议先与维护者沟通确认方向，避免重复工作。

第三步：Fork 本仓库至个人账号，在本地新建功能分支（命名格式为 feature/简述 或 fix/简述），进行代码或文档修改。

第四步：确保新增或修改的代码通过全部单元测试与集成测试，并补充必要的测试用例覆盖变更部分。运行 `npm run lint` 检查代码风格。

第五步：提交 Pull Request 至主仓库的 main 分支，在 PR 描述中关联对应的 Issue 编号，并简要说明变更内容与测试结果。等待维护者 Code Review 后合并。

## 常见问题

**问：为什么部分文章链接在索引中显示但无法正常访问？**  
答：由于 blog.ityiqv.cn 为外部独立站点，其内容可用性不受本项目管理。我们通过每日健康检查脚本自动探测链接状态，并在管理后台标记连续三次不可用的链接为「失效」。如果用户发现链接失效，可通过 Issue 报告，我们将手动复核并从索引中移除或替换为有效存档。

**问：如何提交新的文章链接加入索引？**  
答：项目当前采用批次导入机制，不接受零星新增。但用户可通过 GitHub Issue 提交建议链接，我们会评估后将高质量链接收录至下一批次（第 88/280 批）。提交时请提供文章标题、完整 URL 以及至少一个技术分类标签。

**问：本地开发环境启动后，首页文章列表为空怎么办？**  
答：首次启动时需执行种子数据初始化命令 `npm run seed`，该命令会将本批次 250 条链接的元数据写入 SQLite 数据库。如果仍为空，请检查 database/seeds/ 目录下是否存在有效的 JSON 种子文件，并确认 SQLite3 版本不低于 3.40.0。
