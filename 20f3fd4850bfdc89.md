# TechResource Indexer

TechResource Indexer 是一个面向开发者、技术研究人员与开源贡献者的技术资源聚合与导航系统。该项目并不直接生产原创技术内容，而是通过系统化的方式对互联网上分散的高质量技术文章、教程、案例分析与工程实践记录进行索引、分类与归档，帮助用户在特定技术领域内快速定位到有价值的参考材料。

本项目定位于中大型技术团队的知识管理辅助工具，以及个人开发者构建自身技术知识图谱的起点。通过统一的前端展示面板与结构化的资源分类体系，用户可以在无需记忆大量零散 URL 的前提下，按主题、难度、适用场景等维度检索到所需的外部资源链接。项目核心目标用户包括：需要维护团队技术文档仓库的架构师、从事新技术调研的研发工程师、以及希望建立系统化学习路径的在校学生或转岗开发者。

## 功能概览

**结构化资源索引**：对收录的每一篇外部文章提取标题、摘要、分类标签与原始发布信息，形成统一的元数据登记表，支持后续检索与筛选。

**多维度分类导航**：按照编程语言、框架生态、应用领域、问题类型等多个维度建立分类树，允许用户从不同入口切入资源池。

**全文元数据搜索**：基于内存索引或轻量级全文检索引擎，对已收录资源的标题、描述、标签字段进行快速匹配，返回相关度排序后的结果列表。

**外部链接健康检查**：周期性对已收录的 URL 发起 HEAD 请求，检测链接可用性，标记失效或重定向的资源，保障导航质量。

**资源收藏与批注**：允许注册用户对特定资源添加个人标签、阅读状态与简短批注，便于团队内部分享阅读心得。

**导入导出标准格式**：支持以 JSON、CSV、Markdown 表格等形式批量导入或导出资源清单，方便与其他知识管理工具（如 Notion、Obsidian）进行数据交换。

**访问统计与热度排序**：记录每个外部链接的点击次数与最近访问时间，提供按热度、更新时间、新增时间排序的视图。

## 应用场景

场景一：技术团队新人入职培训。团队负责人可以通过 TechResource Indexer 预先整理好与项目技术栈（如 Go 微服务、React 前端、Kubernetes 运维）相关的必读文章与最佳实践列表，新成员在完成环境搭建后，可直接按照索引顺序进行系统性阅读，缩短上手周期。

场景二：技术调研与选型。当团队需要评估一个新的中间件或框架时，调研负责人可以将网络上收集到的评测报告、性能对比、踩坑记录等链接统一收录到本系统，并添加选型维度的自定义标签。后续评审会议上，所有成员均可通过同一面板查看全部材料，避免信息碎片化。

场景三：个人知识库的补充模块。独立开发者或技术博主可以使用 TechResource Indexer 作为自己博客站点的外链扩展，将阅读过的优秀文章按主题归档，形成公开或私有的学习清单。例如，按月份建立阅读清单，或按难度将资源分为入门、进阶、高级三个层级。

## 快速开始

以下命令演示了如何从 GitHub 克隆项目仓库、安装依赖并启动开发服务器。

```bash
# 克隆仓库到本地
git clone https://github.com/techresource-indexer/techresource-indexer.git

# 进入项目目录
cd techresource-indexer

# 安装项目依赖（使用 npm）
npm install

# 复制环境变量示例文件并填写必要配置
cp .env.example .env

# 启动开发模式服务，默认监听 http://localhost:3000
npm run dev
```

生产环境部署建议使用 `npm run build` 构建静态文件，然后通过 `npm run start` 启动生产服务器。若使用 Docker，可执行 `docker build -t techresource-indexer .` 与 `docker run -p 3000:3000 techresource-indexer`。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 项目运行时基础环境，建议使用 nvm 管理多版本 |
| npm | 9.x 或 10.x | 依赖包管理工具，随 Node.js 一同安装 |
| SQLite | 3.x（内嵌） | 默认元数据存储引擎，无需额外安装，适用于单机部署 |
| Redis | 7.x（可选） | 用于缓存访问统计与会话存储，多实例部署时推荐启用 |
| Nginx | 1.24+（可选） | 生产环境反向代理与静态资源缓存，非必须但建议配置 |
| Git | 2.30+ | 用于克隆仓库及版本管理 |
| curl / wget | 任意版本 | 用于外部链接健康检查脚本的网络请求 |

## 贡献指南

本项目欢迎各类贡献，包括但不限于新增功能、修复缺陷、完善文档以及增加新的资源分类规则。请遵循以下步骤参与贡献。

第一步：查阅项目看板与议题列表。访问 GitHub Issues 页面，查看当前已标记为 "good first issue" 或 "help wanted" 的议题。如计划实现新功能，建议先创建一个议题进行讨论，避免重复工作或偏离项目方向。

第二步：派生仓库并创建功能分支。将主仓库 Fork 至个人账号下，然后克隆到本地。请基于 main 分支创建新的功能分支，分支命名建议采用 `feature/描述` 或 `fix/描述` 格式，例如 `feature/add-json-export`。

第三步：编写代码并添加对应测试。所有新增代码必须包含至少一个正向测试用例和一个边界条件测试用例。确保在提交前运行 `npm run test` 使全部测试通过，并执行 `npm run lint` 保持代码风格一致。

第四步：提交变更并签署开发者原产地证书。提交信息请遵循 Conventional Commits 规范（如 `feat: 增加批量导出为 JSON 格式的接口`）。提交前需阅读并签署 CLA（贡献者许可协议），以确保项目可以合法使用您的贡献。

第五步：发起拉取请求。在 GitHub 上从您的功能分支向主仓库的 main 分支发起 Pull Request。请填写 PR 模板中的检查清单，并关联相关议题编号。维护者将在 3 个工作日内进行审查，必要时会提出修改意见。

## 常见问题

问：项目是否支持对已收录资源进行全文内容搜索，而不仅仅是标题和标签？

答：当前版本仅支持对元数据字段（标题、描述、标签、分类路径）进行检索，不直接对目标页面的 HTML 正文内容建立索引。这是因为本项目定位为导航系统而非全文搜索引擎。若需要全文检索功能，建议将本系统与 Elasticsearch 或 Meilisearch 等专业搜索引擎集成，通过爬取目标页面内容的方式实现。社区已有基于 Puppeteer 的扩展方案，可在开发者文档中查看相关示例。

问：如何批量更新已有资源的分类或标签信息？

答：推荐使用项目内置的 CSV 导入导出功能。首先通过导出功能将现有资源列表导出为 CSV 格式，然后在电子表格软件中批量修改分类或标签列，最后通过导入功能覆盖更新。请注意，导入时需保持 `id` 列不变，否则系统会视为新增资源。对于超过 1000 条资源的批量操作，建议使用 `scripts/batch-update.js` 命令行工具，该工具支持事务性更新，避免部分失败导致的数据不一致。

问：外部链接健康检查的频率和超时时间如何配置？

答：健康检查的调度间隔（默认 24 小时）和单次请求超时时间（默认 5000 毫秒）均通过环境变量配置，详见 `.env.example` 中的 `HEALTH_CHECK_CRON` 与 `HEALTH_CHECK_TIMEOUT` 字段。检查结果会记录在 `health_logs` 表中，并通过管理面板的「链接状态」页面展示。若某链接连续三次检查失败，系统会自动将其标记为「永久失效」并停止后续检查，以节省资源。您也可以手动在管理面板中重新触发单条链接的即时检查。
