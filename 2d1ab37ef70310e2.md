# LinkSphere 技术资源索引系统

LinkSphere 是一个面向开发者、技术研究人员与内容创作者的轻量级技术资源外链汇总与导航系统。该项目定位于对分散在互联网各处的优质技术文章、教程、案例与参考文档进行结构化整理与展示，帮助用户快速定位到特定主题的技术资料，避免在信息海洋中重复检索。

本项目并非一个传统的爬虫或采集系统，而是一个基于人工筛选与主题分类的资源导航工具。其核心目标用户包括：需要查阅特定技术细节的软件工程师、撰写技术报告或论文的研究人员、以及希望系统性学习某一技术栈的初学者。通过统一的条目管理与清晰的分类体系，LinkSphere 能够显著降低技术资料的查找成本，提升知识获取效率。

## 功能概览

**按主题分类索引** 系统将所有收录的资源按照技术领域、编程语言、应用框架等维度进行一级分类，每个分类下包含若干子标签，便于用户按图索骥。

**全文标题与摘要检索** 提供基于关键词的全文检索能力，支持对资源标题、来源站点以及人工添加的摘要信息进行快速匹配，返回相关性排序的结果列表。

**每日更新资源列表** 维护一个动态更新的最新收录板块，按时间倒序展示最近添加的资源链接，方便用户获取新鲜内容。

**收藏与标记暂存** 允许注册用户对感兴趣的资源进行收藏标记，并支持添加个人备注，形成个性化的学习队列。

**RSS 订阅源输出** 为整个资源库或特定分类生成 RSS 订阅源，用户可通过阅读器实时获取新增资源通知，无需频繁访问站点。

**资源有效性检查** 内置一个轻量级的链接状态检查工具，定期对已收录的 URL 进行可访问性探测，并标注失效链接以供维护者清理或更新。

## 应用场景

**技术团队内部知识库构建** 技术团队可以将 LinkSphere 部署为内部知识导航站，将团队沉淀的技术文档、外部参考链接、故障排查记录等统一汇总，形成团队共享的知识入口，减少重复沟通成本。

**开源项目文档外链补充** 开源项目维护者可以在项目的 README 或官方文档中引用 LinkSphere 中相关的技术解析链接，为使用者提供更丰富的上下文参考，而不必在项目仓库内维护大量外部链接。

**教育培训辅助资料整理** 培训机构或高校教师可以围绕课程大纲，在 LinkSphere 中整理对应的扩展阅读材料，学生可以通过分类索引或检索功能快速获取与当前学习进度匹配的课外资料。

**个人技术学习路线规划** 开发者可以按技术主题筛选资源列表，将散落的博客文章、案例解析串联成一条学习路径，配合收藏功能构建个人专属的知识图谱。

## 快速开始

以下步骤将指导您在本机环境快速启动 LinkSphere 服务。

```bash
# 克隆项目仓库
git clone https://github.com/linksphere/linksphere-core.git

# 进入项目目录
cd linksphere-core

# 安装依赖（使用 npm）
npm install

# 配置环境变量，复制示例配置文件
cp .env.example .env

# 初始化本地 SQLite 数据库
npm run migrate

# 启动开发服务器，默认监听 3000 端口
npm run dev
```

启动成功后，在浏览器中访问 http://localhost:3000 即可进入系统首页。管理员后台入口为 /admin，默认管理员账号密码请查看 .env 文件中的初始化配置。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js | 是 | 版本要求 18.x 或更高，推荐使用 LTS 版本 |
| npm 或 yarn | 是 | 包管理工具，npm 版本 8.x 以上 |
| SQLite3 | 是 | 内置支持，无需额外安装，生产环境可切换为 PostgreSQL |
| PostgreSQL（生产可选） | 否 | 生产环境建议使用 PostgreSQL 14+ 作为生产数据库 |
| Redis（缓存可选） | 否 | 启用缓存与会话存储时可使用 Redis 6.x+ |
| Nginx（反向代理可选） | 否 | 生产部署时推荐使用 Nginx 作为前置代理 |
| PM2 或 Docker | 否 | 进程守护或容器化部署工具，生产环境二选一 |

## 贡献指南

本项目的成长离不开社区贡献者的支持。若您希望参与完善资源列表、改进代码或补充文档，请遵循以下流程。

首先，在 GitHub 仓库中提交一个 Issue，简要说明您计划贡献的内容类型（新增资源、Bug 修复、功能增强或文档改进），等待维护者的确认反馈以避免重复工作。

其次，从主干分支 main 或 develop 切出新的功能分支，分支命名规范为 feature/简述主题 或 fix/问题编号。所有代码修改需附带对应的单元测试，并确保现有测试用例全部通过。

第三，遵循项目的代码风格规范（使用 ESLint 与 Prettier 配置），提交信息采用约定式提交格式（type: subject），例如 feat: add batch import for resources 或 docs: update api reference.

第四，在完成本地开发与自测后，发起 Pull Request 到 develop 分支，并在 PR 描述中关联对应的 Issue 编号。PR 需要至少一位核心维护者进行代码审查，审查通过后由维护者合并。

最后，对于资源列表的更新贡献，请确保所提供的 URL 为有效链接，并尽量附带简短的分类标签与摘要说明，以便系统自动归类与检索。新增资源记录需在提交前通过链接有效性检查工具验证。

## 常见问题

问：系统启动后检索功能无法返回结果，应如何排查？

答：请先确认数据库迁移是否完整执行，检索功能依赖于 SQLite 或 PostgreSQL 的全文索引支持。您可以通过执行 npm run reindex 命令手动重建索引。同时检查 .env 文件中 SEARCH_ENGINE 配置项是否与当前数据库类型匹配。若使用 SQLite，请确保系统已安装 FTS5 扩展。

问：资源列表中的链接访问时返回 404 或超时，系统会自动处理吗？

答：LinkSphere 内置的链接有效性检查服务默认每 72 小时对全量资源执行一次探测，并将结果记录在数据库的 status 字段中。您可以在管理后台的「链接状态」面板中查看失效链接列表。对于连续三次检查均失效的链接，系统会将其自动标记为已废弃，不再出现在公开检索结果中。如需手动触发检查，可执行 npm run check:links 命令。

问：如何将系统从 SQLite 迁移至 PostgreSQL 以供生产环境使用？

答：首先在 .env 文件中将 DB_TYPE 修改为 postgres，并配置对应的 DB_HOST、DB_PORT、DB_USER、DB_PASSWORD 与 DB_NAME 参数。然后执行 npm run migrate:prod 运行生产环境迁移脚本。数据迁移工具会在首次启动时自动将 SQLite 中的存量数据导入 PostgreSQL，建议在执行前对原数据库进行备份。迁移完成后，可关闭 SQLite 连接并重启服务。
