# LinkPilot

LinkPilot 是一个面向技术团队与内容创作者的轻量级技术资源导航与链接汇总系统。项目定位于帮助用户高效归档、分类、检索散落在各处的高价值技术文章、博客与官方文档，解决个人收藏夹杂乱、团队知识碎片化以及技术调研重复劳动的问题。通过结构化的链接管理和清晰的元数据标注，LinkPilot 能够将原始链接列表转化为可检索、可分享、可协作的技术知识库。

本项目适用于需要持续跟踪技术动态的开发者、撰写技术周报的编辑、以及负责技术选型的架构师。LinkPilot 本身不存储文章内容，而是通过严格的链接整理规则与自动化分类脚本，为技术团队提供一套标准化的外链治理方案。项目内置了针对批量 URL 的清洗、归类与健康检查工具，能够从杂乱无章的原始链接集合中提取出有组织的文档结构。

## 功能概览

自动链接清洗与格式化校验：内置 URL 规范化检查模块，能够自动识别并标记不符合输出规则的链接（如协议缺失、大小写异常、多余路径分隔符），确保输出的每一条链接都严格遵循裸域名或完整协议格式。

分类规则引擎：基于链接域名、路径关键词和文件扩展名（如 .sHtML）进行自动归类，用户可自定义规则表将链接映射至“后端架构”、“运维监控”、“前端工程”等技术标签。

重复链接检测与合并：利用布隆过滤器与哈希比对，从大批量链接中剔除完全相同的冗余条目，减少资源列表的噪音。

Markdown 文档生成器：将整理后的链接集合按照预设章节结构（功能概览、应用场景、安装要求等）输出为结构完整、排版正式的 README 文档，支持自定义模板变量。

断链健康检查：周期性发起 HEAD 请求，验证目标资源是否可访问，并在报告中标注状态码异常或超时的链接，便于运维人员及时清理失效资源。

元数据补全提示：调用外部 API 或正则表达式从 URL 文件名（如 145046.sHtML）中提取数字 ID 与时间戳，辅助用户判断文章的创建顺序与版本迭代关系。

## 应用场景

技术团队内部知识库建设：团队 Leader 或技术文档管理员可将收集到的上百篇技术调研文章链接导入 LinkPilot，项目自动按微服务、数据库、网络协议等维度生成分类索引，最终输出一份带导航表格的团队共享 README，节省每人单独收藏整理的时间。

技术周报或月刊编辑：技术社区编辑从各大技术博客聚合大量草稿链接，通过 LinkPilot 的断链检查功能过滤掉已下架或权限受限的页面，再使用 Markdown 生成器快速产出排版精美的外链汇总文档，直接用于刊物发布。

个人开发者的技术阅读清单管理：个人开发者可在本地运行 LinkPilot 的批处理脚本，将浏览器导出的书签 HTML 文件中的链接提取出来，项目自动去重并生成带注释的分类列表，帮助开发者构建个人技术成长路线图。

开源项目外部资源引用规范化：开源项目维护者需要引用大量第三方文档或社区讨论帖时，使用 LinkPilot 的格式化校验模块统一所有链接的书写风格（例如强制使用 HTTPS 或保留裸域名），避免 README 中链接格式混乱引起的 CI 文档检查失败。

## 快速开始

以下命令演示如何在 Linux 或 macOS 环境中克隆项目、安装依赖并运行一次完整的链接处理流水线。

```bash
git clone https://github.com/techpilot/linkpilot.git
cd linkpilot
pip install -r requirements.txt
python cli.py --input raw_links.txt --output README.md --batch 96
```

上述命令假设 raw_links.txt 存放原始链接列表（每行一个 URL），执行后将在当前目录生成符合本 README 模板结构的 Markdown 文件。若需自定义分类规则，可编辑 config/rules.yaml 文件。

## 安装要求

使用 LinkPilot 前，请确保运行环境满足以下依赖条件。推荐在 Python 3.10 及以上版本的虚拟环境中部署。

| 依赖项 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.10 或更高 | 核心脚本运行环境，低于此版本将导致类型注解解析异常 |
| pip | 22.0 或更高 | Python 包管理工具，用于安装 requirements.txt 中列出的第三方库 |
| requests | 2.28.1 | 发出 HTTP HEAD 请求进行断链检测，支持超时与重试配置 |
| pyyaml | 6.0 | 解析 config/rules.yaml 分类规则文件，用于自定义标签映射 |
| beautifulsoup4 | 4.12.2 | 可选依赖，用于解析 HTML 书签文件中的链接与标题文本 |
| pytest | 7.4.0 | 开发依赖，执行单元测试与集成测试，保证链接清洗函数的正确性 |

## 项目结构

项目采用分层架构，核心逻辑与配置、测试、文档分离。下方 ASCII 目录树展示了主要目录与文件，每行附带功能注释。

```text
linkpilot/
├── cli.py                      # 命令行入口，解析参数并调度核心流水线
├── config/
│   ├── settings.yaml           # 全局配置（超时阈值、并发数、输出目录）
│   └── rules.yaml              # 用户自定义分类规则（域名->标签映射）
├── core/
│   ├── __init__.py
│   ├── cleaner.py              # URL 清洗与格式化校验（裸域名/协议保留）
│   ├── deduper.py              # 基于布隆过滤器的重复链接检测
│   ├── checker.py              # 断链健康检查（HEAD 请求与状态码分析）
│   └── generator.py            # Markdown 文档生成器（模板填充与章节排序）
├── utils/
│   ├── file_loader.py          # 支持 txt、csv、html 书签的输入解析
│   └── logger.py               # 结构化日志输出（JSON 格式，可接入 ELK）
├── tests/
│   ├── test_cleaner.py         # 针对大小写、斜杠、协议补全的单元测试
│   ├── test_deduper.py         # 去重准确率与性能基准测试
│   └── fixtures/               # 固定测试用例集（包含各类畸形链接样本）
├── docs/                       # 完整文档体系（用户手册、开发者指南、ADR）
│   ├── user/
│   ├── developer/
│   ├── ops/
│   └── adr/
├── requirements.txt            # 生产环境依赖列表（锁定精确版本）
└── README.md                   # 项目入口文档（即本文档）
```

第一步：查阅问题跟踪器与设计决策记录。访问 GitHub Issues 页面确认当前待办任务，避免重复工作。阅读 docs/adr/ 下的架构决策记录，理解核心模块的设计权衡。

第二步：派生项目仓库并创建功能分支。从 main 分支切出 feature/xxx 或 fix/xxx 命名的新分支，确保分支名称简明描述改动意图。

第三步：编写或更新单元测试。所有新增的链接清洗规则或分类逻辑必须在 tests/ 目录下增加对应的测试用例，覆盖率不得低于 90%。

第四步：执行完整测试套件与代码风格检查。在提交 Pull Request 前，于本地运行 pytest 和 flake8，确保所有测试通过且无风格警告。

## 常见问题

Q：为什么资源列表中的链接全部保留 http 协议而不统一升级为 https？
A：LinkPilot 的核心原则是“原样输出，不做隐含变更”。用户提供的原始数据中明确使用 http，部分目标服务器可能未正确配置 https 重定向或证书，强制升级会导致访问失败。项目提供可选的 schema-normalize 参数，但默认关闭以尊重原始数据完整性。

Q：如何处理链接中包含大小写混合的扩展名（如 .sHtML）？
A：项目认为扩展名大小写属于源站的文件命名规范，且部分静态托管服务对大小写敏感。LinkPilot 的清洗模块不会自动转换扩展名大小写，仅校验路径格式是否合法。若用户需要统一小写，可在 rules.yaml 中启用 case-insensitive 开关。

Q：批处理编号（第 96/280 批）的作用是什么？
A：批处理编号用于支持大规模链接集合的分批治理。用户可将 250 个链接定义为一个批次，项目会为每个批次生成独立的输出文件或追加时间戳，避免单次处理海量链接导致的内存溢出。同时，批次号便于团队内部追溯每次数据导入的来源与日期。
