# LinkVault — 技术文章与资源外链聚合系统

LinkVault 是一个面向开发者、技术研究员与内容策展人的轻量级外链资源聚合平台。该项目不生产原创内容，而是系统化地收集、分类、索引来自互联网的高质量技术文章、教程、案例分析与工程实践笔记，帮助技术团队与个人开发者高效检索外部知识资产。

本项目定位为技术资源导航工具，适用于需要长期跟踪外部技术文献、批量整理历史文章链接、或构建内部技术知识库外挂索引的场景。LinkVault 本身不存储文章内容，仅维护链接元数据与分类标签，所有访问均直连源站。

## 功能概览

**结构化链接索引** — 按文章编号、发布时间、所属栏目自动生成索引目录，支持基础检索与筛选。

**批量导入与去重** — 支持从 CSV、JSON 或纯文本列表批量导入 URL，自动检测重复条目并合并标签。

**元数据快照缓存** — 对每个外链记录标题、摘要、爬取时间戳与响应状态码，便于检测链接有效性。

**标签与分类体系** — 允许为每条链接打多级标签（如「后端」「性能优化」「安全」「架构」），支持自定义分类树。

**只读镜像模式** — 对外暴露只读的链接列表 API 与静态页面，适合嵌入团队文档站或仪表盘。

**健康检查定时任务** — 后台定期发起 HEAD 请求验证链接可用性，输出失效链接报告。

## 应用场景

**技术团队内部知识库外链管理** — 研发团队在编写项目文档或周报时，需要引用外部技术博客或官方公告。LinkVault 提供统一的链接入库与检索接口，避免团队成员各自收藏、链接散落各处。

**开源项目 README 引用资源整理** — 开源项目维护者常需要在 README 中列出相关教程、生态工具或案例站点。LinkVault 可快速将批量 URL 转为分类列表，并自动生成 Markdown 格式输出，节省手动整理时间。

**技术资讯聚合与历史归档** — 内容运营人员或技术编辑定期收集特定领域（如云原生、微服务、数据库内核）的文章链接。LinkVault 支持按批次（如第 74/280 批）管理链接集合，便于追溯采集时间和来源。

**个人开发者学习路径规划** — 开发者可将散落在浏览器书签中的技术文章链接统一导入 LinkVault，按主题打标签，形成个人外链学习库，配合健康检查功能及时发现失效资源。

## 快速开始

以下命令将在本地启动 LinkVault 实例，默认监听 8080 端口。

```bash
git clone https://github.com/your-org/linkvault.git
cd linkvault
pip install -r requirements.txt
python app.py --port 8080 --mode standalone
```

若使用 Docker 运行，可执行：

```bash
docker build -t linkvault:latest .
docker run -d -p 8080:8080 -v ./data:/app/data linkvault:latest
```

启动后访问 http://localhost:8080 查看链接列表，初始管理员账号为 admin / changeme，首次登录请修改密码。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9+ | 是 | 核心运行环境，推荐使用 3.10 或 3.11 |
| SQLite 3.35+ | 是 | 默认元数据存储引擎，支持 JSON 字段 |
| requests 2.28+ | 是 | 用于链接健康检查与元数据抓取 |
| beautifulsoup4 4.11+ | 否 | 如需自动提取页面标题与摘要则必须安装 |

| gunicorn 20.1+ | 否 | 生产环境部署时推荐使用，开发模式使用内置 flask server |
| nodejs 16+ | 否 | 仅当需要构建前端管理面板时才需要 |

## 贡献指南

1. 在 GitHub 上 fork 本仓库，创建特性分支（如 feature/batch-importer-optimization），确保分支命名清晰反映改动范围。

2. 本地开发环境建议使用 Python 3.10 虚拟环境（venv 或 conda），运行 `pip install -r requirements-dev.txt` 安装开发依赖，包括 pytest、black、flake8。

3. 提交代码前执行 `make lint` 和 `make test` 确保通过所有静态检查与单元测试，测试覆盖率不低于 80%。

4. 为新增功能编写对应的文档片段（位于 docs/ 目录下相应章节），并在 PR 描述中关联对应 issue 编号。

5. 提交 Pull Request 时请填写模板中的检查清单，包括是否更新了 CHANGELOG、是否添加了迁移脚本、是否兼容现有数据库 schema。

## 常见问题

Q: 导入包含 250 条链接的批次时，页面卡顿或超时怎么办？

A: 推荐使用异步导入模式，在 .env 中设置 `ASYNC_IMPORT=true`，导入任务将放入后台队列，页面立即返回任务 ID，可通过 `/import/status/<task_id>` 轮询进度。对于超过 500 条的批次，建议分割为多个子批次导入。

Q: 健康检查显示大量链接为失效（404/500），但这些链接在浏览器中可正常访问？

A: 部分源站对 HEAD 请求返回非 200 状态码，但允许 GET 请求。可在配置中启用 `HEALTH_CHECK_METHOD=GET` 并设置 `HEALTH_CHECK_TIMEOUT=10`，同时禁用重定向跟随（`ALLOW_REDIRECTS=false`）以避免误判。另外，某些站点会针对非浏览器 User-Agent 返回错误页，请在环境变量中自定义 `USER_AGENT`。

Q: 能否将 LinkVault 部署到内网环境，不连接外网？

A: 可以。LinkVault 核心功能（导入、索引、检索、导出）完全不依赖外网。唯一需要外网的是健康检查与元数据抓取服务，这两项可通过配置 `ENABLE_EXTERNAL_FETCH=false` 关闭。关闭后仅保留本地已缓存的数据，适合纯内网知识库场景。
